Features

Everything you need to talk to databases safely, consistently, and with AI agents in the loop.

Unified API

One interface for MongoDB, PostgreSQL, MySQL, MSSQL, SQLite, and Elasticsearch. Write once, query any backend.

AI Guardrails

Empty filter blocking prevents accidental DELETE *. Unbounded query protection enforces limits. Safe by default.

Self-Correcting Errors

Every StrictDBError includes a fix property with actionable instructions. AI agents can read the fix and retry automatically.

Schema Discovery

describe() lets AI agents learn your tables, columns, and types at runtime. No hardcoded schemas needed.

MCP Server

AI agents talk to databases via the Model Context Protocol. Install strictdb-mcp and connect any MCP-compatible client.

Custom Sanitize Rules

Transform, validate, or reject field values before they reach the database. Define rules once, enforce everywhere.

Supported Databases

StrictDB auto-detects your database from the connection URI. Install the peer dependency for your backend and you're ready to go.

MongoDB Auto-detected

Quick Start

Get up and running with StrictDB in under a minute.

1

Install

Install StrictDB and the peer dependency for your database.

# Core package
npm install strictdb

# Add your database driver (pick one or more)
npm install mongodb        # MongoDB
npm install pg             # PostgreSQL
npm install mysql2         # MySQL
npm install mssql          # MSSQL
npm install better-sqlite3 # SQLite
npm install @elastic/elasticsearch # Elasticsearch
2

Connect

StrictDB auto-detects the backend from your URI.

import { StrictDB } from 'strictdb';

// MongoDB
const db = await StrictDB.create({ uri: 'mongodb://localhost:27017/mydb' });

// PostgreSQL
const db = await StrictDB.create({ uri: 'postgresql://user:pass@localhost:5432/mydb' });

// SQLite
const db = await StrictDB.create({ uri: 'sqlite:///path/to/db.sqlite' });
3

Query

Same API regardless of backend. MongoDB-style filters work everywhere.

// Find one document
const user = await db.queryOne('users', { email: 'alice@example.com' });

// Find many with filters, sorting, and pagination
const active = await db.queryMany('users', {
  filter: { status: 'active', age: { $gte: 18 } },
  sort: { createdAt: -1 },
  limit: 20,
  skip: 0
});

// Count documents
const total = await db.count('users', { status: 'active' });
4

Insert & Update

Write operations with full receipt tracking.

// Insert
const receipt = await db.insertOne('users', {
  name: 'Alice',
  email: 'alice@example.com',
  status: 'active'
});
console.log(receipt.insertedId);

// Update with filter
await db.updateOne('users',
  { email: 'alice@example.com' },
  { $set: { status: 'verified' } }
);

// Delete
await db.deleteOne('users', { email: 'old@example.com' });

API Reference

Every method available on the StrictDB instance. All methods return typed OperationReceipt objects for full auditability.

Read Operations

queryOne(collection, filter)

Find a single document matching the filter. Returns null if not found.

queryMany(collection, options)

Find multiple documents with filter, sort, limit, skip, and projection support.

queryWithLookup(collection, options)

Query with a related collection lookup (like SQL JOIN). Two-query approach for SQL backends.

count(collection, filter)

Count documents matching the filter. Returns a number.

Write Operations

insertOne(collection, doc)

Insert a single document. Returns receipt with insertedId.

insertMany(collection, docs)

Insert multiple documents in a single operation. Returns receipt with insertedCount.

updateOne(collection, filter, update)

Update a single document. Enforces single-row semantics across all SQL dialects (ctid, rowid, LIMIT 1, TOP(1)).

updateMany(collection, filter, update)

Update all documents matching the filter. Supports upsert: true option.

Delete & Transaction Operations

deleteOne(collection, filter)

Delete a single document matching the filter. Single-row enforcement on all backends.

deleteMany(collection, filter)

Delete all documents matching the filter. Blocked by guardrails if filter is empty.

batch(operations)

Execute multiple operations as a batch. Each operation specifies its own collection and action.

withTransaction(fn)

Execute operations within a database transaction. Automatic rollback on error. Supports all dialects including MSSQL.

Filter Syntax

StrictDB uses MongoDB-style filters everywhere. The filter translator converts them to SQL WHERE clauses or Elasticsearch Query DSL automatically.

Comparison Operators

$eq, $ne

Equal / not equal. { status: 'active' } is shorthand for { status: { $eq: 'active' } }.

$gt, $gte, $lt, $lte

Greater than, greater or equal, less than, less or equal. Works with numbers, dates, and strings.

Array & Pattern Operators

$in, $nin

Match any value in array / not in array. { role: { $in: ['admin', 'mod'] } }

$regex, $exists, $size

Pattern matching, field existence check, and array length. Regex complexity is validated to prevent ReDoS.

Logical Operators

$and, $or, $nor, $not

Combine multiple conditions. { $or: [{ status: 'active' }, { role: 'admin' }] }

Translation Example

// This MongoDB-style filter:
const filter = {
  status: 'active',
  age: { $gte: 18 },
  role: { $in: ['user', 'admin'] }
};

// Translates to PostgreSQL:
// WHERE "status" = $1 AND "age" >= $2 AND "role" IN ($3, $4)

// Translates to MySQL:
// WHERE `status` = ? AND `age` >= ? AND `role` IN (?, ?)

// Translates to MSSQL:
// WHERE [status] = @p1 AND [age] >= @p2 AND [role] IN (@p3, @p4)

// Translates to Elasticsearch Query DSL:
// { bool: { must: [{ term: { status: 'active' } }, { range: { age: { gte: 18 } } }, ...] } }

SQL Dialect Support

StrictDB handles the differences between SQL dialects so you don't have to. Every edge case is covered.

MSSQL Pagination

TOP(n) for limit-only queries, OFFSET FETCH for skip+limit, with ORDER BY (SELECT NULL) fallback when no sort is specified.

Single-Row Semantics

updateOne and deleteOne enforce exactly one row: PostgreSQL uses ctid, MySQL uses LIMIT 1, SQLite uses rowid, MSSQL uses TOP(1).

Upsert

UPDATE-then-INSERT pattern via performUpsert(). Works identically across all four SQL dialects.

Projection Exclusion

getExcludedFields() and stripExcludedFields() handle field exclusion as post-processing, since SQL doesn't natively support it.

Transaction Support

All dialects support transactions. MSSQL uses BEGIN TRANSACTION via a dedicated dialect parameter. Automatic rollback on error.

Parameter Styles

PostgreSQL uses $1, MySQL/SQLite use ?, MSSQL uses @p1. StrictDB handles all of it transparently.

Schema & Validation

Register Zod schemas and StrictDB auto-generates DDL, validates at runtime, and provides self-correcting error messages when validation fails.

1

Register a Schema

Define your collection schema with Zod and register it.

import { z } from 'zod';

const userSchema = z.object({
  name: z.string().min(1),
  email: z.string().email(),
  age: z.number().int().positive(),
  role: z.enum(['user', 'admin', 'mod']),
  createdAt: z.date().optional()
});

db.registerSchema('users', userSchema);
2

Auto-Generate SQL DDL

StrictDB introspects the Zod schema and generates CREATE TABLE statements for your SQL dialect.

-- Auto-generated for PostgreSQL:
CREATE TABLE IF NOT EXISTS "users" (
  "name" TEXT NOT NULL,
  "email" TEXT NOT NULL,
  "age" INTEGER NOT NULL,
  "role" TEXT NOT NULL,
  "createdAt" TIMESTAMP
);
3

Auto-Generate ES Mappings

For Elasticsearch, schemas become index mappings automatically.

// Auto-generated mapping:
{
  properties: {
    name: { type: 'text' },
    email: { type: 'keyword' },
    age: { type: 'integer' },
    role: { type: 'keyword' },
    createdAt: { type: 'date' }
  }
}
4

Runtime Validation

Documents are validated before insert/update. Failures include a fix property so AI agents can self-correct.

try {
  await db.insertOne('users', { name: '', email: 'bad' });
} catch (err) {
  console.log(err.fix);
  // "Field 'name' must be at least 1 character. Field 'email' must be a valid email."
}

Guardrails & Sanitization

Safety nets that prevent common mistakes — especially important when AI agents are generating queries.

Empty Filter Protection

Destructive operations (updateMany, deleteMany) with an empty filter are blocked by default. Pass confirm: true to override intentionally.

Unbounded Query Blocking

Queries without a limit are blocked to prevent accidentally loading entire tables. Configurable default limit.

SQL Field Validation

Column names are validated against a whitelist. Prevents SQL injection via dynamic field names from untrusted input.

ES Internal Field Blocking

Elasticsearch internal fields (starting with _) are blocked from user queries to prevent index corruption.

Regex Complexity Validation

Regular expressions in $regex filters are analyzed for catastrophic backtracking patterns (ReDoS). Dangerous patterns are rejected.

Custom Sanitize Rules

Define SanitizeRule objects to transform, validate, or reject field values. Applied via applySanitizeRules() in the StrictDB pipeline before every operation.

// Custom sanitize rule example
const rules = [
  {
    field: 'email',
    transform: (value) => value.toLowerCase().trim(),
    validate: (value) => value.includes('@'),
    message: 'Invalid email address'
  },
  {
    field: 'age',
    validate: (value) => typeof value === 'number' && value > 0 && value < 150,
    message: 'Age must be a positive number under 150'
  }
];

const db = await StrictDB.create({ uri, sanitizeRules: rules });

MCP Server

The strictdb-mcp package exposes StrictDB as an MCP server, letting AI agents talk to databases via standardized tools.

What It Does

AI agents (Claude, GPT, etc.) connect via MCP and get access to database tools — query, insert, update, delete, describe — all with StrictDB's guardrails enforced.

Install & Run

Install globally or as a project dependency. Configure via environment variables.

# Install
npm install strictdb-mcp

# Run (reads STRICTDB_URI from environment)
STRICTDB_URI="postgresql://user:pass@localhost:5432/mydb" npx strictdb-mcp

Available MCP Tools

strictdb_query

Query documents with filters, sorting, pagination.

strictdb_insert

Insert one or many documents.

strictdb_update

Update documents matching a filter.

strictdb_delete

Delete documents matching a filter (guardrails enforced).

strictdb_describe

Discover tables, columns, and types at runtime.

strictdb_count

Count documents matching a filter.

Claude Desktop Configuration

{
  "mcpServers": {
    "strictdb": {
      "command": "npx",
      "args": ["strictdb-mcp"],
      "env": {
        "STRICTDB_URI": "postgresql://user:pass@localhost:5432/mydb"
      }
    }
  }
}

Events & Reconnection

StrictDB emits typed events for connection lifecycle and includes automatic reconnection with exponential backoff.

Typed Events

Listen for connected, disconnected, reconnecting, and reconnected events. Fully typed with TypeScript generics.

db.on('connected', () => console.log('Connected'));
db.on('disconnected', (reason) => console.log('Lost connection:', reason));
db.on('reconnecting', (attempt) => console.log('Retry #' + attempt));
db.on('reconnected', () => console.log('Back online'));

Exponential Backoff

Automatic reconnection with configurable base delay, max delay, max retries, and jitter. Prevents thundering herd on database restarts.

const db = await StrictDB.create({
  uri: 'postgresql://localhost:5432/mydb',
  reconnect: {
    enabled: true,
    baseDelay: 1000,   // 1s initial delay
    maxDelay: 30000,   // 30s max delay
    maxRetries: 10,    // give up after 10 attempts
    jitter: true       // randomize to prevent thundering herd
  }
});
